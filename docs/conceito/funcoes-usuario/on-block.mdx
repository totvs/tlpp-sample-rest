OnBlock roda **uma vez na subida do REST**, antes do mapa de serviços ser publicado. Tudo que não estiver em `block()` continua liberado — o oposto é o [OnAllow](/docs/tlpp/rest/funcoes-usuario/on-allow) (lista branca).

## Quando usar

- Modo manutenção — bloquear APIs temporariamente
- Ocultar endpoints internos da documentação/listagem
- Bloquear verbos específicos (`POST` liberado, `DELETE` bloqueado)

## Onde configurar

A chave `OnBlock` só é lida no **Thread Pool** (não nos Slaves). Exemplo:

```ini
[THREAD_POOL]
UserExits=TP_UEX

[TP_UEX]
OnBlock=U_onBlock
```

## Assinatura e parâmetro

A função recebe um objeto `oList` com métodos para bloquear ou ocultar serviços:

| Método | Ação |
|--------|------|
| `oList:block(endpoint, metodo)` | Bloqueia endpoint + verbo |
| `oList:hide(endpoint, metodo)` | Oculta da listagem (sem bloquear execução direta) |
| `oList:getApiList()` | Lista de APIs disponíveis na subida |
| `oList:getServiceName()` | Nome do serviço HTTP |
| `oList:getThreadPoolName()` | Nome do thread pool |

O retorno é ignorado — devolva `Nil`.

Para cada ação informe **endpoint e método** separadamente. É possível bloquear todos os verbos de um path ou apenas alguns.

## Modo Soft

Se uma exceção ocorrer durante o OnBlock, o AppServer **não reinicia** por padrão (comportamento seguro). Para permitir o start mesmo com falha na lista, configure em `TlppData`:

```ini
[THREAD_POOL]
TlppData={"OnBlockSoft":true}
```

Para lista branca (só o que você permitir fica disponível), use [OnAllow](/docs/tlpp/rest/funcoes-usuario/on-allow) em vez de OnBlock.
